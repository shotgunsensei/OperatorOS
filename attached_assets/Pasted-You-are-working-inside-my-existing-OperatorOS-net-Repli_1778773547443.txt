SSO requirements:
OperatorOS must become the identity provider / launcher for the child apps.

Implement an SSO launch flow:

1. User logs into OperatorOS.
2. User clicks a module/app card.
3. Backend checks:
   - user is authenticated
   - user belongs to the tenant
   - tenant is active
   - module entitlement is active for that tenant
   - user is allowed to access that module
4. Backend creates a short-lived SSO launch token.
5. Backend redirects user to the child app SSO consume URL.

Recommended route:
POST /api/sso/launch/:moduleKey

or if existing routing style differs, use the existing project convention.

The SSO token must include:
- iss: OperatorOS issuer URL
- aud: target module key or target app URL
- sub: OperatorOS user id
- email
- name if available
- tenant_id
- tenant_slug
- tenant_name
- tenant_role
- module_key
- entitlements or plan summary if useful
- iat
- exp, no longer than 5 minutes
- jti unique token id

Security:
Use RS256 signing if practical.

Add env vars:
OPERATOROS_BASE_URL=https://operatoros.net
OPERATOROS_SSO_ISSUER=https://operatoros.net
OPERATOROS_SSO_PRIVATE_KEY=<private key PEM or base64 encoded private key>
OPERATOROS_SSO_PUBLIC_KEY=<public key PEM or base64 encoded public key>

If RS256 is too disruptive for the current stack, implement HS256 using:

OPERATOROS_SSO_SECRET=<long random secret>

But prefer RS256 because child apps only need the public key.

Add a JWKS or public-key endpoint if practical:
GET /api/sso/.well-known/jwks.json

or:
GET /api/sso/public-key
